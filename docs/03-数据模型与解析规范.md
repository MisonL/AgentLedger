# AgentLedger 数据模型与解析规范

- 文档版本：v1.0
- 更新时间：2026-03-01
- 关联文档：`02-软件架构设计.md`

## 1. 设计目标

1. 统一主流 AI CLI 与 AI IDE 会话结构。
2. 保留来源可追溯性（主机、路径、会话 ID、事件偏移）。
3. 兼容 SSH 实时拉取与可选同步缓存两种远程输入。

## 2. 统一领域模型

## 2.1 Source

```ts
interface Source {
  id: string;
  kind: 'local' | 'ssh';
  name: string;
  host?: string;
  basePaths: string[];
  accessMode: 'realtime' | 'sync' | 'hybrid';
  syncCron?: string;          // 可选：定时同步表达式
  syncRetentionDays?: number; // 缓存保留天数
  enabled: boolean;
  createdAt: string; // UTC ISO8601
}
```

## 2.2 Session

```ts
interface Session {
  id: string;                 // AgentLedger 内部 UUID
  sourceId: string;
  clientType: 'cli' | 'ide';
  provider: string;           // 例如: codex/claude/gemini/qwen/kimi/trae/codebuddy/cursor...
  nativeSessionId: string;    // 源会话 ID
  workspace?: string;
  model?: string;
  startedAt?: string;
  endedAt?: string;
  messageCount: number;
}
```

## 2.3 NormalizedEvent

```ts
interface NormalizedEvent {
  id: string;
  sourceId: string;
  sessionId: string;
  clientType: 'cli' | 'ide';
  provider: string;
  eventType: 'user' | 'assistant' | 'tool' | 'meta' | 'usage';
  timestamp: string; // UTC ISO8601

  // 内容
  role?: 'user' | 'assistant' | 'system' | 'tool';
  text?: string;

  // 模型与调用
  model?: string;
  requestId?: string;

  // token 口径
  inputTokens?: number;
  outputTokens?: number;
  cacheReadTokens?: number;
  cacheWriteTokens?: number;
  reasoningTokens?: number;

  // 成本
  costUsd?: number;
  costMode: 'raw' | 'estimated' | 'none';

  // 溯源
  sourcePath: string;
  sourceOffset?: number;
  rawHash: string;
}
```

## 3. 存储模型（SQLite）

## 3.1 逻辑表

1. `sources`：数据源配置。
2. `sessions`：会话主表。
3. `events`：标准化事件明细。
4. `usage_daily` / `usage_monthly`：聚合统计。
5. `usage_calendar`：GitHub 风格日历热力图聚合。
6. `budget_policies` / `budget_snapshots`：预算与阈值。
7. `sync_jobs` / `sync_watermarks`：同步任务与增量水位。
8. `audit_logs`：审计日志。
9. `parse_errors`：解析错误。

## 3.2 建表示例（节选）

```sql
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,
  source_id TEXT NOT NULL,
  provider TEXT NOT NULL,
  native_session_id TEXT NOT NULL,
  workspace TEXT,
  model TEXT,
  started_at TEXT,
  ended_at TEXT,
  message_count INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE UNIQUE INDEX idx_sessions_uni
ON sessions(source_id, provider, native_session_id);

CREATE TABLE events (
  id TEXT PRIMARY KEY,
  source_id TEXT NOT NULL,
  session_id TEXT NOT NULL,
  provider TEXT NOT NULL,
  event_type TEXT NOT NULL,
  role TEXT,
  text TEXT,
  model TEXT,
  timestamp TEXT NOT NULL,
  input_tokens INTEGER,
  output_tokens INTEGER,
  cache_read_tokens INTEGER,
  cache_write_tokens INTEGER,
  reasoning_tokens INTEGER,
  cost_usd REAL,
  cost_mode TEXT NOT NULL,
  source_path TEXT NOT NULL,
  source_offset INTEGER,
  raw_hash TEXT NOT NULL
);

CREATE INDEX idx_events_time ON events(timestamp);
CREATE INDEX idx_events_session ON events(session_id);
```

## 4. 多客户端解析规范

## 4.1 CLI 示例：Claude Code

- 典型路径：`~/.claude/projects/<project>/*.jsonl`
- 关键字段：`message.model`, `usage.input_tokens`, `usage.output_tokens`, `cache_read_input_tokens`。
- 规则：
  1. `assistant` 消息视作主 usage 载体。
  2. 缺失成本时按定价表估算。
  3. 时间字段优先 `timestamp`，缺失则使用文件顺序近似。

## 4.2 CLI 示例：Codex CLI

- 典型路径：`~/.codex/sessions/*.jsonl`
- 关键字段：`event_msg/payload/token_count`。
- 规则：
  1. `token_count` 事件绑定到最近一条 assistant 响应。
  2. 若同一轮多次 token 更新，以最后一次为准。
  3. 解析失败保留原始片段 hash 便于追查。

## 4.3 CLI 示例：Gemini CLI

- 典型路径：`~/.gemini/tmp/*/chats/session-*.json`
- 关键字段：`tokens.input/output/cached/thoughts`。
- 规则：
  1. 将 `cached` 映射为 `cacheReadTokens`。
  2. 将 `thoughts` 映射为 `reasoningTokens`。
  3. 不同版本字段差异通过 schema 版本映射层适配。

## 4.4 IDE 示例：Cursor / VS Code / TRAE / 通义灵码 / CodeBuddy

- 典型来源：IDE 工作区存储、全局会话库、插件本地存储或官方接口。
- 规则：
  1. 首选结构化会话存储（SQLite/JSON）；不可用时解析可追溯日志片段。
  2. 提取会话最小字段：`session_id`、`timestamp`、`model`、`usage`、`workspace`。
  3. 无 token 明细时标注 `data_quality=medium/low` 并走估算口径。

## 5. 远程访问与同步协议

## 5.1 输入

1. `source_id`
2. `since` / `until`（UTC）
3. `mode`：`stats` | `search` | `session_detail` | `sync`
4. `access_mode`：`realtime` | `sync` | `hybrid`
5. `query`（search 模式）
6. `session_id`（detail 模式）

## 5.2 输出

统一返回 JSON Lines：

```json
{"type":"meta","source":"host-a","provider":"codex"}
{"type":"event","session":"xxx","timestamp":"2026-03-01T10:00:00Z","tokens":{"input":100,"output":50}}
{"type":"end","ok":true,"count":123,"freshness":"realtime|cache"}
```

## 5.3 约束

1. 仅只读命令模板。
2. 每次调用设置 `timeout` 与 `max_bytes`。
3. 非 0 退出码必须返回错误结构并落审计。

## 5.4 同步作业元数据

1. `sync_job_id`：单次同步任务唯一 ID。
2. `watermark`：按 source/provider 记录上次成功时间或文件偏移。
3. `sync_result`：`success` | `partial` | `failed`，并记录失败原因。

## 6. 统计口径

## 6.1 统一 token 公式

1. `total_tokens = input + output + cache_read + cache_write + reasoning`
2. 缺失字段按 0 处理，但保留 `data_quality` 标记。

## 6.2 成本公式

1. `cost_raw`：直接使用源日志成本。
2. `cost_estimated`：
   - `input * input_price`
   - `output * output_price`
   - `cache_read * cache_read_price`
   - `cache_write * cache_write_price`
   - `reasoning * reasoning_price`
3. 成本展示默认优先 `cost_raw`，并可切换“仅估算”。

## 6.3 聚合维度

1. 时间：日、周、月。
2. 来源：主机、source、provider。
3. 业务：workspace、model、session。
4. 日历热力图：`date -> tokens/cost/sessions`。

## 7. 去重与幂等

1. 事件指纹：`sha256(source_id + provider + native_session_id + timestamp + raw_fragment)`。
2. 重复写入策略：`ON CONFLICT DO NOTHING`。
3. 聚合作业采用“按时间窗重算 + 原子替换”。
4. 同步作业按 `source_id + watermark` 幂等执行。

## 8. 数据质量策略

1. `quality_level`：`high` | `medium` | `low`。
2. 触发降级条件：缺少时间、缺少会话 ID、usage 字段冲突。
3. 在 UI 明示“估算/不完整”标签。

## 9. 隐私与合规

1. 默认不展示密钥、账号标识、绝对路径用户名。
2. 文本内容可配置“仅索引摘要，不存原文”。
3. 导出前执行字段脱敏管道。

## 10. 测试基线

1. 解析器金样本测试（每种 provider >= 30 份样本）。
2. 跨工具对齐测试（同会话多事件聚合一致性）。
3. 统计快照测试（daily/monthly/session/model）。
4. 日历热力图测试（颜色分级、时区切日、过滤器聚合正确性）。
5. 远程协议契约测试（超时、断连、异常输出、缓存回退）。
