# AgentLedger 环境变量参考

- 文档版本：v2.11
- 更新时间：2026-03-06
- 适用范围：`services/ingestion-gateway`、`services/integration`、`services/puller`、`apps/control-plane`、`clients/agent`

## 1. 使用约定

1. 所有变量名采用全大写下划线。
2. 未特别说明时，默认值仅用于本地开发，不代表生产建议值。
3. 涉及回调鉴权的密钥变量必须在 integration 与 control-plane 两端保持一致。

### 3.1.1 integration 通道格式与事件支持

| 通道       | 出站格式                                    | 支持事件类型      | 说明                                           |
| ---------- | ------------------------------------------- | ----------------- | ---------------------------------------------- |
| `webhook`  | 原样转发上游事件 JSON（`application/json`） | `alert`、`weekly` | 不改写业务字段，接收端按治理事件 schema 解析。 |
| `wecom`    | `text` 模板消息                             | `alert`、`weekly` | 发送企业微信机器人文本模板。                   |
| `dingtalk` | `text` 模板消息                             | `alert`、`weekly` | 发送钉钉机器人文本模板。                       |
| `feishu`   | `text` 模板消息                             | `alert`、`weekly` | 发送飞书机器人文本模板。                       |
| `email`    | SMTP 纯文本邮件                             | `alert`、`weekly` | 按事件生成主题与正文，收件人从事件字段 `email_to` 解析。 |
| `email_webhook` | 标准化邮件 JSON（`event_type/subject/from/to/body/...`） | `alert`、`weekly` | 由下游邮件服务消费并发送，保留原始事件（`event` 或 `event_raw`）。 |
| `ticket`   | 标准化工单 JSON（`event_type/title/summary/context/...`） | `alert`、`weekly` | 由工单系统消费，包含结构化上下文（租户、预算、来源、告警/周报标识）与原始事件。 |

### 3.1.2 orchestration 路由口径（新增）

1. 当治理事件 payload 含 `orchestration.channels` 且 `fallback=false` 时，integration 仅向这些 channels 分发，不再继续套用 `broadcast/severity` 旧路由。
2. 当 `fallback=true` 时，integration 退回 legacy routing；当前 execution log 的 `dispatchMode` 正式口径仅有 `rule` 与 `fallback` 两个值。
3. 当 `dedupeHit=true` 或 `suppressed=true` 且没有可分发 channels 时，integration 不再实际下发，但治理侧仍会写入 execution log 以保留审计与排障线索。

### 3.2.1 callback 签名头约定

| 请求头 | 写入方 | 校验/消费方 | 说明 |
| ------ | ------ | ----------- | ---- |
| `X-Integration-Callback-Secret` | integration | integration 入站 handler、control-plane callback 路由 | 共享密钥鉴权头；control-plane 侧读取 `x-integration-callback-secret`（HTTP 头大小写不敏感）。 |
| `X-Integration-Callback-Source` | integration | control-plane（可选消费） | 回调来源标识，当前值为 `api` 或 `nats`，用于排障与审计。 |
| `X-Integration-Callback-Timestamp` | integration | control-plane callback 路由 | 时间戳头（Unix 秒字符串），参与签名串。 |
| `X-Integration-Callback-Nonce` | integration | control-plane callback 路由 | 随机 nonce，参与签名串并用于窗口内防重放。 |
| `X-Integration-Callback-Signature` | integration | control-plane callback 路由 | 签名头：`HMAC-SHA256(secret, timestamp + "\n" + nonce + "\n" + body)`。 |

| `INTEGRATION_WEBHOOK_TIMEOUT` | `10s` | 回调 HTTP 转发 | integration 向 control-plane 发送单次 callback 请求的超时时间。 |

| `INTEGRATION_RETRY_BASE_DELAY` | `2s` | 回调失败重试 | callback 重试指数退避基线。 |

### 4.1 Open Platform Webhook Replay 变量（新增）

| 变量名 | 默认值 | 必填条件 | 说明 |
| ------ | ------ | -------- | ---- |
| `OPEN_PLATFORM_WEBHOOK_SECRET_KEY` | 空 | 生产环境启用 Webhook secret 加密时建议必填 | control-plane 用于加密/解密 Webhook secret 的主密钥；未配置时会退回开发态 key，仅适合本地调试。 |
| `WEBHOOK_REPLAY_MAX_RETRIES` | `2` | 启用 Webhook replay 时可选 | 单条 replay 事件的最大重试次数，不含首轮发送。 |
| `WEBHOOK_REPLAY_RETRY_BASE_DELAY_MS` | `200` | 启用 Webhook replay 时可选 | replay 失败后的指数退避基线（毫秒）。 |

1. Webhook 事件类型现同时支持旧版 `replay.job.*` 兼容事件与新版 `replay.run.*` 运行级事件；新接入建议优先订阅 `replay.run.*`。

### 4.2 OAuth/OIDC/SSO exchange provider 字段约定（新增）

`AUTH_EXTERNAL_PROVIDERS_JSON` 中，`type=oauth2/oidc/sso` 的 provider 可声明以下扩展字段用于授权码交换：

| 字段 | 必填条件 | 示例 | 说明 |
| ---- | -------- | ---- | ---- |
| `tokenUrl` | 启用授权码交换时建议必填 | `https://idp.example.com/token` | OAuth/OIDC/SSO 授权码交换 token endpoint。 |
| `userInfoUrl` | 可选 | `https://idp.example.com/userinfo` | 交换成功后可选拉取用户信息（如 email/name/sub）。 |
| `clientId` | 启用授权码交换时建议必填 | `agentledger-control-plane` | OAuth 客户端 ID。 |
| `clientSecret` | 机密客户端建议必填 | `corp-oidc-client-secret` | OAuth 客户端密钥，仅服务端消费，不应在前端透出。 |
| `scopes` | 可选 | `["openid", "profile", "email"]` | 请求上游授权范围；建议使用字符串数组。 |

能力边界说明：

1. `GET /api/v1/auth/providers` 仅用于登录入口发现，返回可公开字段；`clientSecret` 等敏感字段不应下发前端。
2. OAuth/OIDC/SSO 授权码交换登录由 `POST /api/v1/auth/external/exchange` 执行，服务端按 provider 配置完成 code -> token（可选 userInfo）并签发平台会话。
3. 本平台不托管上游 IdP 登录页，也不代管上游账号体系与 MFA 流程；上游认证成功后再进入 exchange。

### 4.3 外部断言登录签名规范

`POST /api/v1/auth/external/login` 的签名串采用换行拼接：

```text
providerId + "\n" + externalUserId + "\n" + email + "\n" + tenantId + "\n" + timestamp + "\n" + nonce
```

1. 签名算法：`HMAC-SHA256(secret, canonical_payload)`，输出 64 位十六进制字符串。
2. `tenantId` 可为空字符串（当请求不指定租户时）。
3. 服务端会校验：
   - 时间窗（`AUTH_EXTERNAL_ASSERTION_TTL_SECONDS`）
   - nonce 防重放（同 provider 下 nonce 在窗口内只能使用一次）

| `ARCHIVE_MODE` | `local` | 否 | 归档后端模式。支持 `local`、`object`、`hybrid`；`hybrid` 表示同时写本地和对象存储。 |
| `ARCHIVE_LOCAL_ROOT` | `/data/agentledger/archive/raw` | `ARCHIVE_MODE=local/hybrid` 时建议显式配置 | 本地归档根目录。最终路径格式为 `YYYY/MM/DD/<tenant>/<source>/<session>/<event>_<checksum>.jsonl`；启用 ZSTD 压缩后文件扩展名变为 `.jsonl.zst`。 |
| `ARCHIVE_LOCAL_COMPRESSION` | `none` | 否 | 本地归档压缩方式。支持 `none`/`jsonl`（纯 JSONL）和 `zstd`/`jsonl+zstd`/`jsonl.zst`（先生成 JSONL，再以 ZSTD 压缩落盘）。仅影响本地后端；对象存储仍写 `.jsonl`。 |
| `ARCHIVE_OBJECT_PROVIDER` | - | `ARCHIVE_MODE=object/hybrid` 时必填 | 对象存储提供方。当前支持 `s3`、`oss`。 |
| `ARCHIVE_OBJECT_BUCKET` | - | `ARCHIVE_MODE=object/hybrid` 时必填 | 对象存储 bucket 名称。 |
| `ARCHIVE_OBJECT_PREFIX` | `agentledger/archive/raw` | 否 | 对象键前缀。最终对象键为 `<prefix>/YYYY/MM/DD/<tenant>/<source>/<session>/<event>_<checksum>.jsonl`。 |
| `ARCHIVE_S3_REGION` | 继承 `AWS_REGION` | `ARCHIVE_OBJECT_PROVIDER=s3` 时必填 | S3 区域。未显式设置时回退到 `AWS_REGION`。 |
| `ARCHIVE_OSS_ENDPOINT` | - | `ARCHIVE_OBJECT_PROVIDER=oss` 时必填 | OSS Endpoint，例如 `oss-cn-hangzhou.aliyuncs.com`。 |
| `ARCHIVE_OSS_ACCESS_KEY_ID` | 继承 `ALIBABA_CLOUD_ACCESS_KEY_ID` | `ARCHIVE_OBJECT_PROVIDER=oss` 时必填 | OSS AccessKey ID。 |
| `ARCHIVE_OSS_ACCESS_KEY_SECRET` | 继承 `ALIBABA_CLOUD_ACCESS_KEY_SECRET` | `ARCHIVE_OBJECT_PROVIDER=oss` 时必填 | OSS AccessKey Secret。 |
| `ARCHIVE_OSS_SECURITY_TOKEN` | 继承 `ALIBABA_CLOUD_SECURITY_TOKEN` | 否 | 使用 STS 临时凭证时可选，未设置则按长期凭证模式初始化 OSS client。 |

推荐示例：

```bash
# 本地 JSONL + ZSTD
export ARCHIVE_MODE=local
export ARCHIVE_LOCAL_ROOT=/data/agentledger/archive/raw
export ARCHIVE_LOCAL_COMPRESSION=zstd

# 本地 + S3 双写
export ARCHIVE_MODE=hybrid
export ARCHIVE_LOCAL_COMPRESSION=zstd
export ARCHIVE_OBJECT_PROVIDER=s3
export ARCHIVE_OBJECT_BUCKET=agentledger-archive
export ARCHIVE_S3_REGION=ap-southeast-1
```

## 5. DATABASE_URL 依赖说明

1. `services/integration`：当前告警分发与回调转发链路不强依赖 `DATABASE_URL`。
2. `services/ingestion-gateway`：当前不强依赖 `DATABASE_URL`。
3. 强依赖 `DATABASE_URL` 的服务：`services/normalizer`、`services/analytics`、`services/governance`、`services/puller`、`services/archiver`。
4. `apps/control-plane`：未配置 `DATABASE_URL` 时会回退内存模式；生产环境建议显式配置 PostgreSQL。

## 6. 回调链路一致性对照表

| 对照项                      | integration 侧                                                                                           | control-plane 侧                             | 校验脚本                                                                              |
| --------------------------- | -------------------------------------------------------------------------------------------------------- | -------------------------------------------- | ------------------------------------------------------------------------------------- |
| Stream/Subject/Durable 绑定 | `INTEGRATION_CALLBACK_STREAM` / `INTEGRATION_CALLBACK_SUBJECT`(`TOPIC`) / `INTEGRATION_CALLBACK_DURABLE` | -                                            | `scripts/check-callback-stream-binding.sh`                                            |
| 转发目标 URL                | `CONTROL_PLANE_BASE_URL` + `INTEGRATION_CALLBACK_PATH`                                                   | 路由 `/api/v1/integrations/callbacks/alerts` | `scripts/test-callback-chain-targeted.sh`                                             |
| 回调鉴权 Secret             | header `X-Integration-Callback-Secret`                                                                   | 校验 `x-integration-callback-secret`         | `scripts/test-callback-chain-targeted.sh`、`scripts/e2e-governance-callback-chain.sh` |
| 回调来源标识                | header `X-Integration-Callback-Source`                                                                   | 用于日志与审计定位来源（`api`/`nats`）       | `scripts/test-callback-chain-targeted.sh`、`scripts/e2e-governance-callback-chain.sh` |
| 回调签名与防重放            | header `Timestamp/Nonce/Signature`，`INTEGRATION_CALLBACK_SIGNATURE_TTL`                                 | 校验签名、时间窗、nonce 去重                 | `scripts/test-callback-chain-targeted.sh`、`scripts/e2e-governance-callback-chain.sh` |
| 回调时间窗                  | `INTEGRATION_WEBHOOK_TIMEOUT` / `INTEGRATION_RETRY_*` / `INTEGRATION_CONSUMER_ACK_WAIT`                 | 幂等占位固定窗口 `2m`（代码常量）            | `scripts/test-callback-chain-targeted.sh`、`scripts/e2e-governance-callback-chain.sh` |

### 6.1 回调运维检查项

1. 发布前执行 `bun run check:callback-stream-binding`，确认 stream/subject/durable 绑定无冲突。
2. 核对 integration 与 control-plane 的 `INTEGRATION_CALLBACK_SECRET` 一致，避免出现持续 `401`。
3. 校验 `INTEGRATION_CALLBACK_SIGNATURE_TTL` 两端一致，并确认 `Timestamp/Nonce/Signature` 头可达。
4. 校验时间窗组合：`INTEGRATION_RETRY_MAX_DELAY >= INTEGRATION_RETRY_BASE_DELAY`，且建议 `INTEGRATION_CONSUMER_ACK_WAIT > INTEGRATION_WEBHOOK_TIMEOUT`。
5. 执行 `bun run test:callback-chain-targeted` 与 `bun run test:e2e-governance-callback-chain`，确认回调链路可达且可回归。
6. 观察 integration 日志中的 callback `retry/dlq` 与 control-plane `401/4xx/5xx`，出现持续异常时优先排查 secret、签名与目标 URL。

### 6.2 integration 真实消费分发 E2E

建议额外执行：

```bash
bun run test:e2e-integration-routing
```

当前覆盖：

1. `alert` 在 `orchestration.channels` 存在时仅向指定 channels 分发。
2. `fallback=true` 时回退 legacy severity/broadcast routing。
3. `suppressed=true` 时不再继续对外分发。
4. `weekly` 在携带 orchestration 时同样按 override 分发。
5. callback consumer 会把 NATS 消息带签名头转发到 control-plane callback URL。
