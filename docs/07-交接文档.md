## 3. 关键业务目标（必须保持一致）

1. 统一读取 Codex/Claude Code/Gemini CLI 会话。
2. 统一提供 token/cost 统计（daily/monthly/session/model）。
3. 支持本地 + 远程主机数据来源。
4. 支持会话检索、可视化与预算限额跟踪。

## 4. 不可改变的核心约束

1. 远程主机数据获取支持三种模式：`realtime` / `sync` / `hybrid`（默认 `realtime`）。
2. 同步仅允许单向只读缓存，不允许回写远程源。
3. 系统默认只读采集，不对源目录写入。
4. 技术栈优先：Bun + TypeScript。

## 5. 已定技术路线

1. 后端：Bun + Hono + SQLite（WAL）。
2. 前端：React + TanStack Query。
3. 搜索：SQLite FTS5（MVP）。
4. 远程访问：Bun.spawn 调用 ssh，支持实时读取和可选同步缓存。

## 7. MVP 实施建议（接手执行顺序）

## 7.1 第一阶段：工程底座（1 周）

1. 初始化 Bun Monorepo。
2. 创建 `apps/api`、`apps/web`、`packages/core`、`packages/connectors`。
3. 建立 SQLite 迁移机制与基础表。
4. 建立统一错误码、日志格式、配置结构。

交付标准：

1. API 与 Web 可本地启动。
2. 数据库可自动初始化。
3. 基础健康检查接口可用。

## 7.2 第二阶段：采集与解析闭环（1~2 周）

1. 实现 LocalConnector（CLI P0 + IDE P0，含国产 Qwen Code/TRAE IDE）。
2. 实现 SshRealtimeConnector（只读命令白名单）。
3. 实现 SshSyncConnector（手动/定时同步 + 增量水位）。
4. 实现多客户端 Parser + Normalizer。
5. 完成 events/sessions 入库。

交付标准：

1. P0 客户端金样本解析正确率 >= 99%。
2. 远程读取/同步失败可诊断（超时、权限、路径）。

## 7.3 第三阶段：统计与查询（1~2 周）

1. 实现 daily/monthly/session/model 聚合任务。
2. 实现 GitHub 风格日历热力图聚合与 API。
3. 实现会话搜索与过滤 API。
4. 实现 Dashboard、Sessions、Sources 页面。

交付标准：

1. 统计口径与样本核对误差 < 1%。
2. 关键查询 P95 满足既定目标。

## 7.4 第四阶段：预算与导出（1 周）

1. 实现预算阈值规则。
2. 实现告警记录。
3. 实现 CSV/JSON 导出。

交付标准：

1. 阈值告警可稳定触发。
2. 导出内容可复核且带来源追溯字段。

## 8. 接手风险清单

1. 上游日志格式变化导致解析失败。
2. 远程 SSH 网络抖动或离线导致实时查询失败。
3. 成本口径不一致导致报表争议。
4. 数据量增大后搜索与聚合性能下降。

建议缓解：

1. 解析层做 schema 版本探测和回退逻辑。
2. 远程调用必须有超时、重试、限流与缓存回退。
3. UI 明确标识“原始成本/估算成本”。
4. 对高频查询做缓存和索引优化。

## 9. 质量与验收门槛

1. P0 需求完成率 100%。
2. 解析正确率 >= 99%。
3. 统计误差 < 1%。
4. 安全高危问题为 0。
5. 关键流程 E2E 全通过。
