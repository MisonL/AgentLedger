# AgentLedger 交付计划与验收策略

- 文档版本：v2.5
- 更新时间：2026-03-06
- 关联文档：`01-功能需求规格说明.md`、`02-软件架构设计.md`、`13-环境变量参考.md`

## 1. 交付目标

在不牺牲可追溯性与可运维性的前提下，分阶段交付可运行、可验证、可审计的企业级 AI 使用治理平台。

## 2. 里程碑规划

| 里程碑      | 周期  | 交付重点                               | 退出标准（DoD）                                               |
| ----------- | ----- | -------------------------------------- | ------------------------------------------------------------- |
| M1 工程底座 | W1    | Monorepo、基础 API/Web、脚本体系       | `lint/test/build` 可在本地与 CI 稳定执行                      |
| M2 采集解析 | W2-W3 | P0 Connector + 统一事件模型 + 解析链路 | P0 客户端最小采集闭环打通，失败重试可观测                     |
| M3 统计检索 | W4    | usage 聚合、热力图、sessions 检索      | usage 与 sessions 基础 API/UI 可用，关键路径有测试            |
| M4 治理回调 | W5    | budgets、alerts、integration callback  | governance -> integration -> control-plane 链路通过脚本化回归 |
| M5 稳定发布 | W6    | 跨平台构建、文档完备、发布检查         | 多架构构建通过（含 `darwin-arm64`），三平台最小运行冒烟通过   |

## 3. 验收策略

### 3.1 分层测试

1. 单元测试：解析逻辑、预算规则、回调安全校验。
2. 集成测试：source/sessions/pricing/alerts/identity-binding 等 API 与存储协同。
3. 联调测试：callback 链路、支持矩阵与配置绑定一致性。
4. 发布回归：核心页面可访问、核心脚本可执行、关键接口可响应。

### 3.2 发布门槛

1. 功能门槛：MVP 范围内接口与页面达到“最小可用”。
2. 质量门槛：`lint`、`test`、`build`、`check:support-matrix`、`sdk:verify`、`sdk:test`、`check:callback-stream-binding`、`test:e2e-governance-routing`、`build:agent-cross && verify:agent-cross` 全绿（其中 `test` 需覆盖 `apps/control-plane` 的 identity 绑定用例）。
3. 治理链门槛：发布候选版本至少跑一次真实治理链 E2E，覆盖 `fallback / dedupe / suppressed / fail-open / weekly`。
4. 配置门槛：回调链路变量、签名头约定与时间窗配置与代码绑定一致，禁止文档漂移。
5. 文档门槛：README 与 `docs/05`、`docs/09`、`docs/13`、`docs/15`、`docs/releases/*` 同步更新。

### 3.3 Release 工作流严格门禁（`v*` tag）

1. `bun run lint`
2. `bun run test`
3. `bun run build`
4. `bun run check:support-matrix`
5. `bun run sdk:verify`
6. `bun run sdk:test`
7. `bun run check:callback-stream-binding`
8. `bun run test:e2e-governance-routing`
9. 任一步失败立即终止 `pre-release-gate` Job，不进入发布阶段。
10. 仅当前 8 步全通过时，才进入 `build-and-publish` 执行 `bun run build:agent-cross && bun run verify:agent-cross`，生成 `SHA256SUMS.txt` 并发布产物。

### 4.1 Coverage 门禁阈值

1. Go 核心包阈值：
   - `services/ingestion-gateway` 行覆盖率 `>= 70%`
   - `services/puller` 行覆盖率 `>= 70%`
   - `services/integration` 行覆盖率 `>= 75%`
2. `apps/control-plane` 门槛：`All files` 行覆盖率 `>= 80%`。
3. 统一入口：`bun run test:coverage`（采集 + 阈值校验）。

## 5. 风险与缓解

| 风险               | 影响               | 缓解策略                             |
| ------------------ | ------------------ | ------------------------------------ |
| 客户端日志格式漂移 | 采集与解析失败     | 版本探测 + 兼容解析 + 回退策略       |
| 回调配置不一致     | 告警闭环中断       | 固化变量命名 + 脚本化一致性校验      |
| 统计口径争议       | 报表可信度下降     | 原始值与估算值并行展示，保留计算口径 |
| 文档滞后代码       | 联调误判、上线风险 | 文档与脚本联动校验，评审清单强制核对 |

## 6. 发布前检查清单

1. `bun run lint`
2. `bun run test`
3. `bash ./scripts/test-identity-targeted.sh`（identity 绑定改动时必跑）
4. `bun run build`
5. `bun run check:support-matrix`
6. `bun run sdk:verify`
7. `bun run sdk:test`
8. `bun run check:callback-stream-binding`
9. `GOV_E2E_DATABASE_URL=... bun run test:e2e-governance-routing`
10. `bun run build:agent-cross && bun run verify:agent-cross`
11. `bun run test:callback-chain-targeted`
12. `bun run test:e2e-integration-routing`
13. `bun run test:e2e-governance-callback-chain`
14. `GOV_E2E_DATABASE_URL=... bun run test:e2e-governance-integration-downstream`
15. `bun run smoke:fr505`
16. 更新本次 `docs/releases/*` 发布说明，写清 replay 新事件、orchestration routing 行为与兼容性说明。
17. 核对 callback 签名头：`X-Integration-Callback-Secret`、`X-Integration-Callback-Source`。
18. 核对 callback 签名头：`X-Integration-Callback-Timestamp`、`X-Integration-Callback-Nonce`、`X-Integration-Callback-Signature`。
19. 核对 callback 时间窗配置：`INTEGRATION_CALLBACK_SIGNATURE_TTL`、`INTEGRATION_WEBHOOK_TIMEOUT`、`INTEGRATION_RETRY_*`、`INTEGRATION_CONSUMER_ACK_WAIT`。
20. 核对 `docs/13` 中回调变量与代码一致。
21. 核对 `README` 的目录、门禁、里程碑描述与当前仓库一致。
22. 发布时使用 `v*` tag 触发 `Release` 工作流并检查产物与 `SHA256SUMS.txt`。

补充说明：

- `test:e2e-governance-routing` 默认强制要求 `GOV_E2E_DATABASE_URL` 已配置。
- GitHub Release 工作流需在仓库 Secrets 中提供 `GOV_E2E_DATABASE_URL`，否则 `pre-release-gate` 会直接失败。

### 6.1 回调运维检查项

1. 上线前确认 integration 与 control-plane 的 `INTEGRATION_CALLBACK_SECRET` 一致，避免 `401` 鉴权失败。
2. 上线前确认 `INTEGRATION_CALLBACK_SIGNATURE_TTL` 两端一致，并抽样检查 `Timestamp/Nonce/Signature` 头。
3. 用 `bun run check:callback-stream-binding` 校验 callback stream/subject/durable 绑定，避免 consumer 绑定冲突。
4. 以 `bun run test:callback-chain-targeted` 做快速回归，以 `bun run test:e2e-governance-callback-chain` 做闭环校验。
5. 检查 callback 失败路径日志与 DLQ 增量；若重试持续上升，优先排查目标 URL、secret、签名和时间窗参数组合。
